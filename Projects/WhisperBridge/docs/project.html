WhisperBridge

Bridges a Vent-Axia Svara fan's BLE GATT interface to WiFi and MQTT, making it controllable from Home Assistant without modifying the fan.

Overview

WhisperBridge is an ESP32 firmware that acts as a wireless bridge between the Vent-Axia Svara bathroom fan and a home automation network. The Svara fan exposes a BLE GATT interface for control but has no native WiFi or MQTT support. WhisperBridge solves this by sitting on the same network as a Home Assistant instance, subscribing to an MQTT command topic, and translating incoming boost commands into the required BLE GATT write sequence on the fan.

When a boost command arrives via MQTT (triggered by a Home Assistant automation, voice assistant, or the device's own web UI), WhisperBridge connects to the fan by MAC address, authenticates with a PIN, sends the boost payload, and disconnects — all on a dedicated FreeRTOS task so the WiFi and MQTT event loop is never blocked. State feedback (ON / OFF) is published back to MQTT so Home Assistant can track whether a boost is in progress.

First-time setup uses a captive-portal AP mode: connecting to the WhisperBridge-Setup WiFi network opens a browser page that scans for available networks and saves credentials to NVS. On subsequent boots the device connects in station mode, advertises via mDNS as whisperbridge.local, and registers itself with Home Assistant automatically through MQTT discovery.

BLE Command Sequence

Every boost command triggers this seven-step BLE sequence, executed on the ble_boost FreeRTOS task. The task blocks until the sequence completes, then signals the main loop to publish the OFF state.

1
Connect to the fan by MAC address (AA:BB:CC:DD:EE:FF) using NimBLE at maximum TX power (ESP_PWR_LVL_P9)
2
Discover the Auth service by UUID e6834e4b-7b3a-48e6-91e4-f1d005f564d3
3
Write the PIN bytes e3 14 62 05 to the auth characteristic with a confirmed write (awaits GATT acknowledgement)
4
Wait 200 ms (vTaskDelay) for the fan to process authentication before proceeding
5
Discover the Command service by UUID c119e858-0531-4681-9674-5a11f0e53bb4
6
Write the boost payload 01 60 09 84 03 to the command characteristic with a confirmed write
7
Disconnect and release the NimBLE client; _running flag clears; main loop publishes OFF to MQTT state topic
info
FreeRTOS task isolation keeps the device responsive. The entire BLE sequence — connect, authenticate, command, disconnect — blocks for several seconds. Running it on a dedicated FreeRTOS task (ble_boost, priority 5, 8 KB stack) means the Arduino loop task continues servicing the MQTT client, OTA updates, and REST requests without any stall. The task wakes on xTaskNotifyGive and blocks again on ulTaskNotifyTake between commands.

Hardware

WhisperBridge runs on a standard ESP32 DevKit V1 development board. All primary communication is wireless (WiFi + BLE). The board needs power via USB or a 5 V supply and should be placed within BLE range of the fan. An optional status LED on GPIO 2 lights up while a BLE boost sequence is running — useful during installation and debugging.

Placement matters. NimBLE is configured at maximum TX power (ESP_PWR_LVL_P9) because the fan may be across the room or behind a wall. Wiring

All primary communication is wireless. The only required physical connections are the USB power supply to VIN and GND. An optional status LED on GPIO 2 (with a 330 Ω current-limiting resistor) lights HIGH while the BLE boost sequence is running. It can be omitted or disabled by commenting out STATUS_LED_PIN in config.h.

Software / Firmware

The firmware is a PlatformIO/Arduino project targeting esp32dev. It is structured into three translation units — main.cpp owns WiFi, the web server, and OTA; ble.cpp encapsulates the NimBLE GATT client and FreeRTOS task; mqtt.cpp manages the PubSubClient connection and Home Assistant discovery. A native PlatformIO environment runs Unity unit tests for the MQTT payload parser without requiring hardware.

First Boot — WiFi Setup

If no WiFi credentials are stored in NVS, the device starts in AP mode broadcasting WhisperBridge-Setup (password: whisperbridge). Connect to this network — any HTTP request is redirected to the captive portal at http://10.0.0.1/setup.html. Select a network from the scan results, enter the password, and submit. The device saves the credentials to NVS namespace "whisper" (keys: ssid, pass) and schedules a restart via the s_pendingRestart flag — the actual ESP.restart() is deferred to loop() to avoid calling it from within an AsyncWebServer callback running on the lwIP task.

MQTT Reference

All topics are scoped by the device ID — the last 3 bytes of the MAC address in lowercase hex (e.g., 2cb120). The device ID is printed to serial on boot and available via GET /api/deviceinfo. PubSubClient is configured with a 512-byte buffer and reconnects every 5 seconds when disconnected.

Topics

TopicDirectionPayloadRetainNotes
whisperbridge/{id}/boost Subscribe ON or 1 No Triggers the BLE boost sequence. OFF / 0 are ignored — boost is one-shot, not a toggle.
whisperbridge/{id}/boost/state Publish ON / OFF Yes ON published immediately (optimistic) when command received. OFF published by loop() when _running transitions false after BLE completes.
homeassistant/switch/whisperbridge_{id}_boost/config Publish JSON discovery payload Yes HA MQTT auto-discovery config. Published on every MQTT connection so HA always has the latest config.

BLE Reference

The Vent-Axia Svara fan exposes two GATT services over BLE. Authentication must precede any command write. WhisperBridge uses h2zero/NimBLE-Arduino ^1.4.2 in GATT client mode — it initiates all connections; the fan acts as a peripheral/server. All writes use confirmed write (with response) to ensure the fan has acknowledged the value.

info
Re-trigger guard. BleBoost::trigger() checks the _running atomic flag before calling xTaskNotifyGive — if a sequence is already in progress the new trigger is silently dropped. The REST POST /api/boost endpoint returns HTTP 409 in this case. Both _running and _lastSuccess are std::atomic<bool> for safe cross-task reads from the main loop.

REST API

The web server (ESPAsyncWebServer on port 80) serves both the web UI from LittleFS and a JSON REST API. Available endpoints depend on whether the device is in station mode or AP/captive-portal mode.

Station Mode Endpoints

EndpointMethodResponseNotes
/api/status GET {"running":bool,"ble_ok":bool,"ip":"…","rssi":-42} Polled by the web UI every 2 s. running = BLE sequence in progress; ble_ok = result of last completed sequence.
/api/boost POST {"ok":true} or {"error":"already running"} (409) Triggers BLE boost. Returns 409 if a sequence is already running.
/api/deviceinfo GET {"id":"2cb120","url":"http://whisperbridge.local"} Available in both station and AP mode.

AP / Captive Portal Mode Endpoints

EndpointMethodResponseNotes
/api/networkdata GET {"scanning":bool,"networks":[{"ssid":"…","rssi":-60,"secure":true},…]} Deduplicated results sorted by signal strength (strongest first). Kicks off a fresh async scan for the next request.
/api/networkset POST {"ok":true} Body: {"ssid":"MyNet","password":"secret"}. Saves to NVS and sets s_pendingRestart=true; restart deferred to loop() to avoid calling it from the lwIP task.
/api/deviceinfo GET {"id":"2cb120","url":"http://whisperbridge.local"} Available in both modes.
