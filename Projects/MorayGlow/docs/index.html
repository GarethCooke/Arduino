MorayGlow

WiFi-connected RGB LED strip controller built on the Seeed XIAO ESP32-S3 — REST API, live WebSocket, and Home Assistant MQTT auto-discovery, served from a captive-portal-capable device with no cloud dependency.

Overview

MorayGlow is a compact WiFi-connected RGB LED strip controller built around the Seeed XIAO ESP32-S3. It drives a 12 V 5050 RGB strip through three IRLML6344TR N-channel MOSFETs, with smooth hardware PWM at 1 kHz delivered by the ESP32's LEDC peripheral across three independent 8-bit channels. An MP1584 buck converter steps the 12 V supply down to 5 V to power the XIAO, so the only external supply needed is a single 12 V DC adapter.

The device exposes a REST API and a live WebSocket feed, meaning any browser on the local network can control power, colour, and animation mode in real time. The web UI is bundled in LittleFS and served directly from the XIAO — no cloud intermediary, no mobile app. Full Home Assistant integration is handled through MQTT auto-discovery: on every broker reconnect the device publishes its configuration to homeassistant/light/<device-id>/config, registering itself as a light entity with RGB colour support.

On first boot, or after a factory reset triggered by a 5-second button hold, the device launches a captive-portal access point so WiFi credentials can be configured from any phone or laptop. Once connected, it registers a MAC-derived mDNS hostname (morrayglow-XXXXXX.local) and an mDNS device-discovery endpoint lets a single web UI tile browse and control all MorayGlow units on the network simultaneously.

State Change Sequence

1
State change arrives via REST POST, incoming WebSocket message, or MQTT command on <device-id>/command
2
Handler writes to shared globals: ledOn, ledColor (hex string), cycleMode
3
applyLedState() drives GPIO1–GPIO3 via LEDC hardware PWM — RGB channels update immediately
4
broadcastState() pushes JSON state to all connected WebSocket clients so every open browser tab updates in real time
5
mqttPublishState() publishes a retained message to <device-id>/state — Home Assistant receives the new light state
info
Single unified state path. Every control source — REST, WebSocket, or MQTT — converges on the same three global variables before calling applyLedState(). There is no per-source state; after any change, all clients see a consistent view. This means an MQTT command from Home Assistant is instantly reflected in any open browser tabs, and vice versa.

warning
12 V strip requires external MOSFETs. The XIAO's GPIO pins operate at 3.3 V and source only a few milliamps — far too weak to switch a 12 V RGB strip directly. See the Wiring section for the circuit overview.

Wiring / Schematic

A single 12 V DC supply powers both the RGB strip and (via the MP1584 buck converter) the XIAO. GPIO1–GPIO3 drive the MOSFET gates for the Red, Green, and Blue channels respectively. An optional status LED on GPIO4 uses a 330 Ω current-limiting resistor to GND. The tactile button on GPIO5 uses the internal pull-up — pressing it pulls the pin LOW.

The RGB strip's common anode connects to 12 V; each colour cathode connects to the MOSFET drain. Gate resistors (100 Ω recommended) between the GPIO pins and MOSFET gates improve switching performance and protect against ringing.

Simplified schematic — not to scale. Gate resistors (100 Ω recommended) between GPIO pins and MOSFET gates omitted for clarity. Full BOM and netlist in docs/Hardware/.

Firmware

The firmware is built with PlatformIO targeting the Arduino framework on the Seeed XIAO ESP32-S3. Three PlatformIO environments are defined: xiao_esp32s3 for USB flashing, xiao_esp32s3_ota for over-the-air updates via ArduinoOTA (password: password), and native for running C++ unit tests on the host without a device. The web UI assets in data/ are uploaded separately to LittleFS using the uploadfs target.

State is held in three globals defined in main.cpp and extern'd into webserver.cpp and mqtt.cpp: ledOn, ledColor (a #rrggbb hex string), and cycleMode. The state.h header provides three inline helpers — stateToJson(), rgbToHex(), and parseMqttCommand() — that are guarded by #ifndef ARDUINO so they compile on the native platform for unit tests using Unity via PlatformIO.

First-Time Setup

1

Edit MQTT configuration

Open MorrayServer/include/config.h and set your MQTT broker IP address and credentials before flashing.

cppcontent_copy
#define MQTT_HOST     "192.168.1.100"   // your broker IP
#define MQTT_PORT     1883
#define MQTT_USER     "mqtt_user"
#define MQTT_PASSWORD "mqtt_pass"
2

Build and flash firmware via USB

Connect the XIAO via USB-C. On Windows, pio is not on PATH — invoke the full executable path.

powershellcontent_copy
cd MorrayServer
& "$env:USERPROFILE\.platformio\penv\Scripts\pio.exe" run -e xiao_esp32s3 -t upload
3

Upload web UI to LittleFS

The web UI files in data/ are a separate upload target. Run this after every change to HTML/CSS/JS files.

powershellcontent_copy
& "$env:USERPROFILE\.platformio\penv\Scripts\pio.exe" run -e xiao_esp32s3 -t uploadfs
4

Configure WiFi via captive portal

On first boot (no saved credentials), the device starts AP MorrayGlow-XXXXXX. Connect to it and visit http://10.0.0.1 to enter your WiFi credentials. The device restarts into station mode.

5

OTA updates (after WiFi is configured)

Once the device is on your network, flash wirelessly using the OTA environment. The device hostname is printed to serial on boot.

powershellcontent_copy
# Flash firmware over WiFi
& "$env:USERPROFILE\.platformio\penv\Scripts\pio.exe" run -e xiao_esp32s3_ota -t upload --upload-port morrayglow-a1b2c3.local

# Flash filesystem over WiFi
& "$env:USERPROFILE\.platformio\penv\Scripts\pio.exe" run -e xiao_esp32s3_ota -t uploadfs --upload-port morrayglow-a1b2c3.local
6

Run native unit tests (no device needed)

Unit tests for state.h logic (JSON serialisation, hex conversion, MQTT command parsing) run on the host via the native environment using Unity.

powershellcontent_copy
& "$env:USERPROFILE\.platformio\penv\Scripts\pio.exe" test -e native

Client Dev Server

The MorrayClient/test-server/ Node.js mock server mirrors the firmware's REST and WebSocket API exactly. Use it to develop the web UI without a physical device.

bashcontent_copy
cd MorrayClient/test-server
npm install   # first time only
npm start     # → http://localhost:3000
warning
OTA requires Windows Firewall rules. ArduinoOTA requires the ESP32 to connect back to the host on an ephemeral port. Windows Firewall must allow inbound TCP connections for %USERPROFILE%\.platformio\penv\Scripts\python.exe. IoT network isolation (common on mesh routers) will block the OTA callback — move the device to the main network or use USB flashing.

MQTT Topics

All topics use the runtime device ID derived from the last 3 bytes of the MAC address. Example device ID: morrayglow-2cb120.

DirectionTopicRetainedPurpose
Subscribe <device-id>/command — Receives commands from Home Assistant or any MQTT client
Publish <device-id>/state Yes Current device state; also used as LWT (offline = {"on":false})
Publish homeassistant/light/<device-id>/config Yes HA auto-discovery payload; published on every broker reconnect

MQTT Payload Format

The MQTT command payload uses the Home Assistant JSON light schema, which differs from the REST API format:

json — MQTT command (subscribe)content_copy
// Subscribe: morrayglow-2cb120/command
{
  "state": "ON",
  "color": { "r": 255, "g": 0, "b": 0 },
  "cycle": false
}
json — MQTT state (publish)content_copy
// Publish: morrayglow-2cb120/state
{
  "on": true,
  "color": "#ff0000",
  "cycle": false
}
info
Schema difference between REST and MQTT. The REST API uses {"on":bool} and {"color":"#rrggbb"}. The MQTT command topic uses the HA JSON light schema: {"state":"ON"} (string) and {"color":{"r":255,"g":0,"b":0}} (RGB object). Parsing is handled by parseMqttCommand() in state.h.

Key Architecture Notes

  • state.h is header-only with inline functions and #ifndef ARDUINO guards so it compiles on both the ESP32 and the native host (for unit tests).
  • The three globals (ledOn, ledColor, cycleMode) are defined once in main.cpp and extern'd into webserver.cpp and mqtt.cpp — no RTOS synchronisation is needed because ESPAsyncWebServer callbacks run in the same task context.
  • WiFi credentials are persisted to EEPROM using the WiFiConfig struct from IguanaLib. The struct is zero-filled on factory reset, causing isValid() to return false on the next boot and triggering AP mode.
  • The WiFi scan is always asynchronous (WiFi.scanNetworks(true)). The result is cached and served from the next /api/networkdata request; a fresh scan is kicked off immediately after each result is consumed.
